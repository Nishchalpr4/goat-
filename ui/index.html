Query Pipeline — 8 Stages

Every query flows through an 8-stage pipeline. Each stage transforms the raw text into a structured, scored, explainable result. Click any stage to see internals.

1

Parse

▶
Raw query string→ QueryParser.parse()→ ParsedQuery

The parser first normalizes the text (Unicode NFC, case folding) via TextNormalizer, then tokenizes it using field-aware rules for identifiers, narrative text, and schema terms.

  • Period extraction — detects fiscal periods like FY2024, Q3 2025, H1 using regex patterns
  • Entity mentions — matches ticker patterns ([A-Z]{1,5}) and company name fragments
  • Metric mentions — identifies financial metrics like "gross margin", "revenue", "EPS"
  • Comparison detection — flags "vs", "compare", "versus" operators
  • Negation detection — spots "not", "without", "excluding" modifiers

Output: ParsedQuery with tokens, periods[], entity_mentions[], metric_mentions[], comparison_type, negation flag.

2

Detect Intent

▶
ParsedQuery→ IntentDetector.detect()→ IntentResult

Classifies the query into one of 7 intents, each routing to a different retrieval tier:

  • find_evidence — factual lookup ("What is AAPL's revenue?") → Tier A
  • compare — multi-entity comparison ("AAPL vs MSFT margins") → Tier B
  • summarize — document summary ("Summarize GOOGL 10-K") → Tier B
  • trend — time-series analysis ("NVDA revenue trend") → Tier B
  • screen — multi-company screening ("Companies with >30% margin") → Tier B
  • explain — causal reasoning ("Why did NVDA decline?") → Tier C
  • compute — derived calculation ("Calculate AAPL ROIC") → Tier C

Each intent carries a confidence score and maps to one of three retrieval tiers (A = fast, B = precise, C = reasoning).

3

Resolve Entities

▶
Entity mentions→ EntityResolver.resolve_batch()→ ResolutionResult

Each entity mention undergoes 4-stage resolution to find its canonical company ID:

  • Stage 1 — Exact ticker match: Checks if "AAPL" maps directly to a known ticker
  • Stage 2 — Exact lexicon match: Searches the full synonym lexicon for exact surface forms
  • Stage 3 — Trigram ticker match: Fuzzy matching using trigram similarity (threshold ≥ 0.3) against tickers — catches "GOOG" → "GOOGL"
  • Stage 4 — Trigram alias match: Fuzzy matching against all aliases — catches "Apple" → AAPL, "Microsoft" → MSFT

Returns: canonical IDs, company IDs, metric IDs with confidence scores at each stage.

4

Expand Query

▶
Raw query + ResolutionResult→ QueryExpander.expand()→ ExpandedQuery

The 7-step expansion pipeline enriches the query for maximum recall:

  • Step 1 — Tokenize: Re-tokenize normalized query
  • Step 2 — Resolve entities: Inject canonical forms ("AAPL" → "Apple Inc.")
  • Step 3 — Expand abbreviations: "FCF" → "Free Cash Flow", "EPS" → "Earnings Per Share"
  • Step 4 — Phrase mining: Detect multi-word financial terms ("net operating income")
  • Step 5 — Schema linking: Map to XBRL concepts ("net income" → us-gaap:NetIncomeLoss)
  • Step 6 — Synonym injection: Add domain synonyms ("revenue" ↔ "top line" ↔ "net sales")
  • Step 7 — Final assembly: Combine all expanded terms into retrieval-ready query
5

Generate Plan

▶
Parsed + Intent + Expansion→ QueryPlanner.plan()→ QueryPlan

The planner builds an executable QueryPlan based on the detected tier:

  • Tier A (fast): Lexical (BM25) + Semantic (ANN) + RRF fusion — no reranking, no graph
  • Tier B (precision): + Cross-encoder reranking of top-50 candidates
  • Tier C (reasoning): + Graph traversal for causal/relational context expansion

Plan includes: retrieval_text, company_ids, periods, doc_types, and per-stage configs (top_k sizes, fusion method, reranker model).

6

Execute Retrieval

▶
ExpandedQuery + Plan→ HybridRetriever.retrieve()→ RetrievalResult

The hybrid retrieval engine runs a 5-stage sub-pipeline:

  • Lexical retrieval — BM25/FTS via Postgres tsvector + ts_rank_cd with section boosts (MD&A 1.3×, Risk Factors 1.2×)
  • Semantic retrieval — Dual-channel dense ANN via Qdrant. Channel 1: narrative embedding (pure text). Channel 2: entity+schema conditioned embedding (prefixed with Company|Ticker|Sector|DocType|Section)
  • Reciprocal Rank Fusion (RRF) — Combines all ranked lists: RRF(d) = Σ 1/(k + rank_i(d)) with k=60. Deduplicates across channels
  • Reranking (Tier B/C) — Cross-encoder reranking of top-50 fused candidates via Voyage/Cohere models
  • Graph expansion (Tier C) — Traverse knowledge graph for related entities, causal facts, provenance chains

Each hit gets a ScoringBreakdown: lexical_score, semantic_score, trigram_similarity, fused_score, rerank_score, recency_boost, entity_match_boost, final_score.

7

Graph Expansion (Tier C)

▶
Resolved entity IDs→ GraphTraverser.expand_for_query()→ GraphExpansionResult

Only activated for Tier C queries (explain, compute). Traverses the multi-zone knowledge graph:

  • Walks from resolved entities through Entity → Knowledge → Data zones up to depth 2
  • Collects related entities (subsidiaries, competitors, executives)
  • Gathers facts (financial data points), claims (extracted statements), events (M&A, partnerships)
  • Follows provenance edges to source documents for auditability
  • Returns GraphExpansionResult with augmented context for answer generation
8

Explain Results

▶
RetrievalResult→ QueryExplainer.explain_result()→ list[ExplainedHit]

Final stage annotates each retrieval hit with human-readable scoring explanations:

  • Breaks down the ScoringBreakdown into natural language: "Ranked #2 with lexical score 0.78 (matched 'revenue', 'margin' via BM25), semantic score 0.85, fused via RRF = 0.82, reranked to 0.91"
  • Highlights which terms matched (lexical), which embeddings were closest (semantic), and how fusion combined scores
  • Attaches provenance: source document, section, company, filing date
  • Enables full auditability — every score can be traced back to its origin
Ingestion Pipeline — 8 Stages

Documents (10-K/10-Q filings, transcripts, news) are processed through 8 stages before becoming queryable.

1

Parse Document

▶
Raw text + metadata→ create_parser(doc_type)→ ParseResult(Document, sections)

Specialized parsers for each document type: SEC filings (10-K, 10-Q, 8-K), earnings transcripts, press releases, news articles. Extracts structural information like sections (Risk Factors, MD&A, Financial Statements), tables, and exhibits.

2

Validate

▶
Document→ DataValidator.validate_document()→ ValidationResult

Sanity checks: required fields present, valid encoding, content length within bounds, no duplicate doc_ids, metadata consistency. Flags issues but allows ingestion to continue with warnings.

3

Chunk

▶
Document→ DocumentChunker.chunk_document()→ list[Chunk]

Splits documents into retrieval-optimized chunks (default: 512 tokens, 64 overlap). Uses structural segmentation — detects section boundaries (headers, numbered items) and avoids splitting mid-sentence. Each chunk carries its section name, position offsets, and parent document reference.

4

Embed (Dual-Channel)

▶
list[Chunk]→ EmbeddingManager.embed_chunks()→ list[DualEmbeddingResult]

Each chunk gets two separate embeddings:

  • Narrative embedding — pure text via text-embedding-3-small (1536 dims) for general semantic similarity
  • Entity+Schema conditioned — text prefixed with [Company|Ticker|Sector|DocType|Section] via text-embedding-3-large (3072 dims) for entity-aware retrieval

This dual strategy ensures "AAPL revenue" matches differently from "MSFT revenue" even when the text is similar.

5

Index Vectors

▶
Chunks + Embeddings→ VectorStore.upsert()→ HNSW index in Qdrant

Stores embeddings into two Qdrant collections: goat_narrative and goat_entity_schema. Uses HNSW indexing (m=16, ef_construct=200) for fast ANN search. Payload indexes on company_id, sector, doc_type, year enable metadata-filtered search.

6

Index Full-Text (Postgres)

▶
Chunks + Document→ PostgresStore.fts_index()→ tsvector + trigram index

Creates PostgreSQL full-text search entries with tsvector representations and GIN indexes for BM25 ranking. Also creates pg_trgm GIN indexes for trigram fuzzy matching — enabling misspelling tolerance and partial-match queries.

7

Build Graph

▶
ParseResult + Chunks + Companies→ GraphBuilder.build_*()→ Nodes & edges in 5 zones

Populates the multi-zone knowledge graph: creates company nodes, links to sectors/exchanges, extracts financial facts as data nodes, creates claim/event nodes from parsed content, maps lexical entries to alias/synonym nodes, and records provenance chains from source documents.

8

Record Provenance

▶
All intermediate results→ ProvenanceRecord creation→ W3C PROV triples in graph

Records full W3C PROV-DM provenance: which system version processed the document (wasAttributedTo), which extraction run generated chunks (wasGeneratedBy), which embedding model was used (wasAssociatedWith), and which source document each chunk derives from (wasDerivedFrom). Enables complete auditability and reproducibility.

Lexical Semantics Subsystem

A first-class processing layer ensuring "revenue", "top line", "net sales", and us-gaap:Revenues all resolve to the same canonical concept.

▶

The 7-step lexical-semantic pipeline runs on every query and every ingested document:

1. Normalize
Unicode NFC, case folding, accent removal, whitespace cleanup
2. Tokenize
Field-aware: IDENTIFIER, NARRATIVE, SCHEMA, PERIOD, CURRENCY types
3. Resolve
4-stage entity resolution with trigram similarity (≥0.3 threshold)
4. Schema Link
Map to XBRL concepts (us-gaap:NetIncomeLoss, etc.)
5. Abbreviations
FCF → Free Cash Flow, EPS → Earnings Per Share (46 entries)
6. Synonyms
Revenue ↔ Top Line ↔ Net Sales via canonical ID lookup
7. Condition
Prepend entity context for conditioned embeddings

Module Breakdown

Query Pipeline

8-stage orchestrator: Parse → Intent → Resolve → Expand → Plan → Retrieve → Graph → Explain. Routes queries through tiered retrieval based on complexity.

parser.py · intent.py · planner.py · explainer.py · pipeline.py

Ingestion Pipeline

8-stage document processor: Parse → Validate → Chunk → Embed → Index Vectors → Index FTS → Build Graph → Record Provenance. Handles SEC filings, transcripts, news.

parsers.py · loader.py · validator.py · edgar.py · pipeline.py

Lexical Semantics

First-class NLP subsystem: Unicode normalization, field-aware tokenization, 4-stage entity resolution, XBRL schema linking, synonym expansion. Ensures "revenue" = "top line" = "net sales".

normalizer.py · tokenizer.py · entity_resolver.py · schema_linker.py · query_expander.py · synonym_manager.py

Hybrid Retrieval

5-stage retrieval: Lexical (BM25) + Semantic (dual ANN) + RRF Fusion + Reranking + Graph Expansion. Three tiers: A (fast), B (precision), C (reasoning).

lexical.py · semantic.py · fusion.py · reranker.py · hybrid.py

Dual Embeddings

Two embeddings per chunk: narrative (text-embedding-3-small, 1536d) for general similarity, entity+schema conditioned (text-embedding-3-large, 3072d) for entity-aware retrieval.

models.py · manager.py · chunker.py

Graph RAG

5-zone knowledge graph (Entity, Data, Knowledge, Lexical, Provenance). NetworkX MultiDiGraph backend. Community detection via Louvain. Depth-2 traversal for Tier C queries.

zones.py · builder.py · traversal.py · community.py

Storage Layer

Four backends: PostgreSQL (FTS + trigram + canonical data), Qdrant (HNSW vector search), NetworkX (multi-zone graph), DuckDB (analytics + evaluation).

postgres.py · vector_store.py · graph_store.py · duckdb_store.py

Operations & Reporting

Version tracking, latency/quality monitoring, Precision@k/NDCG evaluation harness. Terminal, JSON, and CSV exporters for results.

versioning.py · monitoring.py · evaluation.py · terminal.py · json_export.py · csv_export.py
Data Models

Company

canonical_id, name, identifiers[] (ticker/ISIN/CIK/LEI), aliases[], metadata (sector, industry, market_cap), financials (income/balance/cashflow), management[], tags[]

Document & Chunk

Document: doc_id, title, content, metadata (company, doc_type, filing_date). Chunk: chunk_id, text, offsets, section, company context, dual embedding IDs, provenance.

Financials

Income: revenue, COGS, gross/operating/net margins. Balance: assets, liabilities, equity, ratios. Cash Flow: OCF, capex, FCF, buybacks.

Retrieval & Scoring

RetrievalResult: hits[], tier, fusion_method, latencies. ScoringBreakdown: lexical/semantic/trigram/fused/rerank/final scores per hit.
